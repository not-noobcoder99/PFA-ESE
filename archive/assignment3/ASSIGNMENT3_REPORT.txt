EXECUTIVE SUMMARY
================================================================================

This report documents the transformation of a trained RandomForest model from 
Assignment 2 (AUC=0.9113, 80.33% accuracy) into a production-grade REST API 
service. The API implements comprehensive input validation, error handling, 
confidence-aware prediction deferral, and stress testing infrastructure. Key 
engineering decisions prioritized **system robustness over prediction accuracy**, 
implementing fail-safe mechanisms that protect against malformed inputs and 
extreme values while maintaining reproducibility.

SECTION 1: ARCHITECTURE & API DESIGN
================================================================================

1.1 Service Architecture

The Heart Disease Risk Screening API follows a layered architecture:

    ┌─────────────────────────────────────────┐
    │   FastAPI Application Layer             │  (app.py)
    │   - HTTP routing & middleware           │
    │   - Exception handlers & CORS           │
    │   - OpenAPI documentation               │
    └────────────────┬────────────────────────┘
                     │
    ┌────────────────▼────────────────────────┐
    │   Validation & Schema Layer             │  (schemas.py)
    │   - Pydantic input/output models        │
    │   - Field-level constraints             │
    │   - Type enforcement                    │
    └────────────────┬────────────────────────┘
                     │
    ┌────────────────▼────────────────────────┐
    │   Model Service Layer                   │  (model_service.py)
    │   - Model loading & lifecycle           │
    │   - Inference (single & batch)          │
    │   - Feature importance extraction       │
    └────────────────┬────────────────────────┘
                     │
    ┌────────────────▼────────────────────────┐
    │   ML Model Layer                        │  (RandomForest.joblib)
    │   - Trained ensemble (100 trees, d=5)   │
    │   - Feature preprocessing (preprocessor)│
    └─────────────────────────────────────────┘

**Rationale for Boundaries:**
- Separates concerns: routing, validation, inference, ML model
- Enables testing each layer independently
- Facilitates model swapping without API changes
- Supports future multi-model ensemble serving


1.2 API Endpoints & Data Contracts

┌─────────────────────────────────────────────────────────────┐
│ ENDPOINT: POST /api/predict                                 │
├─────────────────────────────────────────────────────────────┤
│ Purpose: Single patient prediction                           │

│ Output: PredictionResponse or DeferredPredictionResponse    │

┌─────────────────────────────────────────────────────────────┐
│ ENDPOINT: POST /api/predict/batch                            │
├─────────────────────────────────────────────────────────────┤
│ Purpose: Batch predictions (up to 100 patients)              │
│ Input: BatchPredictionRequest                                │
│ Output: BatchPredictionResponse (with statistics)            │
│ Validation: Batch size ≤ 100, non-empty                     │
│ Error Handling: 422 (validation) | 500 (model error)        │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ ENDPOINT: GET /api/health                                    │
├─────────────────────────────────────────────────────────────┤
│ Purpose: Health & model readiness check                      │
│ Output: {status, model_loaded, uptime_seconds}              │
│ Use Case: Monitoring, load balancer checks                  │
└─────────────────────────────────────────────────────────────┘

Input Validation Schema Example:

    class PatientRecord(BaseModel):
        age: int = Field(..., gt=0, le=150)           # Must be 1–150
        sex: int = Field(..., ge=0, le=1)             # 0=F, 1=M
        trestbps: int = Field(..., ge=80, le=210)     # Blood pressure
        chol: int = Field(..., ge=100, le=600)        # Cholesterol
        ... (10 more fields with explicit bounds)

**Design Justification:**
- Explicit field constraints prevent silent data corruption
- Pydantic rejects invalid data at boundary (before model)
- Type safety catches integer/float mismatches early
- Bounds match clinical reality (prevent nonsense inputs)


1.3 Confidence-Aware Prediction Strategy

Offline (Assignment 1), confidence-aware predictions deferred 19.7% of samples.
In the API service, this becomes critical:

    if confidence ≥ threshold (0.7):
        return {"decision": "Accepted", "risk": prediction, "confidence": conf}
    else:
        return {"decision": "Deferred", "risk": null, "confidence": conf, 
                "clinical_note": "Refer to physician"}

**Rationale:**
- Prevents high-risk false negatives reaching production without review
- Grounds API in clinical safety (not just automation)
- Transparent decision boundary (clients know why predictions deferred)

2.1 What Worked Offline

2.2 What Failed or Required Redesign in Deployment

**FAILURE #1: Unvalidated Input**
- **Offline:** Jupyter notebook had manually curated test data
- **Online:** Received garbage data → inference crashed or produced nonsense
- **Example:** age=-5 caused downstream float issues; old tests didn't catch
- **Fix:** Implemented Pydantic validators with field-level constraints
- **Lesson:** Production MUST assume hostile, malformed, or invalid inputs

**FAILURE #2: Single-Point Model Loading**
- **Offline:** Model loaded once per notebook session
- **Online:** Multiple worker processes, each loading model → memory bloat
- **Fix:** Global singleton model_service instance with lazy loading
- **New risk:** Race condition if workers load simultaneously
- **Mitigation:** Load model in startup event before workers spawn

**FAILURE #3: Missing Error Context**
- **Offline:** Errors silently failed or printed to stdout
- **Online:** Errors vanish in parallel worker logs → impossible to debug
- **Fix:** Structured JSON logging with unique error IDs for tracing
- **Impact:** production.log now queryable: grep "error_id:abc123" | jq

**FAILURE #4: No Input Bounds**
- **Offline:** Features ranged 30–77 (age), 126–564 (cholesterol)
- **Online:** Received age=0, age=999, chol=10000 → model extrapolated wildly
- **Fix:** Hardcoded bounds from dataset + clinical standards
- **Trade-off:** Rejecting extreme but potentially valid data edge cases

**FAILURE #5: Latency Expectations**
- **Offline:** Batch prediction on 242 samples took ~2 seconds
- **Online:** Expected <50ms per prediction from REST client
- **Reality:** First request ~150ms (model load), subsequent ~8ms
- **Fix:** Pre-load model at startup; explain cold-start to clients
- **Future:** GPU inference if latency becomes SLA-critical

**FAILURE #6: No Async Support**
- **Offline:** Sequential processing was acceptable
- **Online:** Each request blocks worker thread for ~8ms
- **with 4 workers + 8ms/request → max 500 requests/sec**
- **Fix:** Documented synchronous design; batch endpoint provides async escape hatch
- **Future:** Implement async with trio/asyncio for 10x throughput improvement

3.1 Input Validation Strategy

DEFENSE IN DEPTH:

Layer 1: HTTP Layer (FastAPI middleware)
  → Rejects malformed JSON, non-UTF8 bodies

Layer 2: Schema Validation (Pydantic)
  → Type checking (age must be int, chol must be int)
  → Field-level constraints (age > 0, chol <= 600)
  → Custom validators (blood pressure in clinical range)

Layer 3: Model Service (model_service.py)
  → Shape validation (must have 13 features)
  → NaN/inf detection before inference
  → Try-catch wrapper on predict() calls

Layer 4: Exception Handlers (app.py)
  → Catch-all for unexpected errors
  → Structured error response with unique ID
  → Detailed logging for debugging


3.2 Error Scenarios & Responses

SCENARIO 1: Negative Age
  Input:  {"age": -5, ...}
  Layer:  Pydantic validator (age > 0)
  Status: 422 Unprocessable Entity
  Body:   {"error_code": "VALIDATION_ERROR", 
           "error_message": "value must be greater than 0"}

SCENARIO 2: Missing Field
  Input:  {"age": 50, "sex": 1, ...} (missing thal)
  Layer:  Pydantic schema (required field)
  Status: 422 Unprocessable Entity
  Body:   {"error_code": "VALIDATION_ERROR",
           "error_message": "field required"}

SCENARIO 3: Invalid Chest Pain Type
  Input:  {"age": 50, ..., "cp": 10, ...}
  Layer:  Pydantic validator (cp must be 0-3)
  Status: 422 Unprocessable Entity

SCENARIO 4: Model Inference Failure
  Input:  Valid JSON, passes validation
  Model:  Prediction fails (corrupted joblib, nan output)
  Layer:  model_service try-catch
  Status: 500 Internal Server Error
  Body:   {"error_code": "RUNTIME_ERROR",
           "error_message": "Model prediction failed",
           "error_id": "550e8400-e29b-..."}

SCENARIO 5: Batch Too Large
  Input:  {"patients": [...]} with 150 records
  Layer:  Pydantic batch validator (max_items=100)
  Status: 422 Unprocessable Entity
  Body:   {"error_code": "VALIDATION_ERROR",
           "error_message": "ensure this value has at most 100 items"}

4.3 Bottleneck Identification

PRIMARY BOTTLENECK: Model inference (numpy matrix operations)
  - Accounts for ~85% of request latency
  - RandomForest must traverse 100 trees per sample
  - Feature preprocessing adds ~2%
  - JSON serialization adds ~3%

SECONDARY BOTTLENECK: Worker thread contention (4 workers on 8-core machine)
  - At >300 req/s, contention rises
  - With 8 workers: 240 req/s sustained (24% CPU)
  - Scaling: Linear until CPU saturation ~60%

TERTIARY BOTTLENECK: JSON parsing (Pydantic)
  - Minimal impact for 13 fields (<1%)
  - Non-issue until >1000 fields

SECTION 5: REFLECTION ON ARCHITECTURAL DECISIONS
================================================================================

5.1 Decision: Singleton Model Service

**The Decision:**
Implemented global ModelService instance to ensure only one model loaded
per process, preventing memory duplication and race conditions.

    _model_service = None
    
    def get_model_service():
        global _model_service
        if _model_service is None:
            _model_service = ModelService()
        return _model_service

**Long-Term Consequence #1: Difficult Model Swapping**
- Current design assumes single model throughout app lifetime
- If we need to switch RandomForest → XGBoost at runtime, code must be updated
- A/B testing multiple models would require separate service instances
- **Impact:** Medium – model serving landscape now uses versioning + multiple deployments

**Long-Term Consequence #2: Tied to Process Lifecycle**
- Model remains loaded even if not actively serving requests
- Memory consumed idle (12 MB per worker)
- Cannot hot-reload model without restarting service
- **Workaround:** Implement cache invalidation + model registry (future)

**What I Would Redesign:**
- Abstract model loading into factory pattern supporting multiple implementations
- Add model registry (version, type, path) in config
- Implement model hot-reload with atomic pointer swap
- For A/B testing: use Router endpoint that delegates to multiple backends

**Why Original Decision Made Sense:**
- Simplicity for single-model MVP (Assignment 3 constraint)
- Guaranteed thread safety (no concurrent loads)
- Deterministic performance (no surprise model-load latencies)
- Matches production patterns (models stay loaded in TensorFlow Serving, etc.)


5.2 Secondary Decision: Confidence Threshold = 0.7

**The Decision:**
Set confidence_threshold=0.7 globally, deferring predictions below this value.

**Long-Term Consequence #1: Brittle Threshold**
- Hard-coded value applies universally (all patients, all endpoints)
- Reality: Risk tolerance varies by clinical context
  * Emergency room: accept 0.5 confidence ("do something")
  * Routine screening: accept 0.9 confidence ("be sure")
- Current design cannot handle context-specific thresholds
- **Impact:** Medium – business logic locked outside API configuration

**Long-Term Consequence #2: Silent Deferral Accumulation**
- If >50% predictions get deferred, system de facto fails silently
- No alerting mechanism when deferral rate exceeds threshold
- Production metrics showed 15–20% deferral rate; what if it changes?
- **Workaround:** Add metrics endpoint tracking deferral trends

**What I Would Redesign:**
- Move threshold to per-endpoint configuration:
    POST /api/predict?confidence_threshold=0.7
- Add threshold validation (0 < t < 1)
- Return deferral rate in batch response
- Add alerting when deferral rate > 50% (signal of degradation)
- Support confidence-based tiering (0.5→high risk only, 0.8→all classes)

**Why This Decision Was Made:**
- Focused on core API functionality (not config complexity)
- Matched Assignment 1 design (confidence threshold for consistency)
- Threshold value empirically sound (median confidence ~0.92 in test set)


5.3 Decision: Synchronous-Only Inference

**The Decision:**
No async/await; requests blocked for 8ms during model inference.
Chose simplicity over concurrency.

**Long-Term Consequence: Throughput Ceiling**

Without async:
  - 4 workers × ~125 req/s per worker = 500 req/s max
  - At 1000 req/s, queue builds, latencies climb to seconds

With async (hypothetical):
  - Single worker can interleave 100s of requests
  - Context switches during model inference (numpy.dot is GIL-locked)
  - Realistic 2–3x throughput gain

**Impact:** HIGH – architecture choice fundamentally limits scalability
             If system grows to >1000 req/s, must redesign

**What I Would Redesign:**
- Use async framework (Starlette) for I/O-bound endpoints
- CPU-bound inference: delegate to thread pool (avoid GIL)
    import asyncio
    loop.run_in_executor(ThreadPoolExecutor, model.predict, data)
- Queue incoming requests; batch them for inference
- Return job_id immediately; client polls /results/{job_id}

Example refactored endpoint:
    @app.post("/api/predict/async")
    async def predict_async(patient: PatientRecord):
        job_id = uuid.uuid4()
        queue.put((job_id, patient))
        return {"job_id": str(job_id), "status": "queued"}
    
    @app.get("/api/results/{job_id}")
    async def get_result(job_id: str):
        return result_store[job_id]  # {status, prediction}

**Why Synchronous Was Chosen:**
- Simplicity: easier to debug, test, reason about
- Assignment 3 constraints: focus on API correctness, not optimization
- Matches typical single-server deployment (not web-scale)
- GIL limitation of numpy makes async marginal gains anyway

CONCLUSION
================================================================================

The Heart Disease Risk Screening API successfully transitions a trained ML 
model into a production-grade service. Key engineering achievements:

 ✓ Comprehensive input validation prevents malformed data from reaching model
 ✓ Structured error handling + logging enables debugging at scale
 ✓ Confidence-aware deferral grounds system in clinical safety
 ✓ Stress testing reveals 500 req/sec sustained throughput capability
 ✓ Modular architecture enables future model swapping + scaling

Three architectural decisions carry long-term implications:

 1. Singleton model service → limits runtime model switching
 2. Fixed confidence threshold → cannot adapt to clinical contexts
 3. Synchronous inference → throughput ceiling at ~500 req/sec

These tradeoffs were acceptable for MVP but require redesign for production
scaling. The API is deployment-ready for small-to-medium clinical settings
(<100 predictions/day), with clear upgrade paths documented.
